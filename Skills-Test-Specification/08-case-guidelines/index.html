各类 Case 编写要求

Trigger Case

Section titled “Trigger Case”

用于验证 Skill 是否应该被触发。

要求至少包含：

  • 一个正例：应该触发该 Skill；
  • 一个反例：不应该触发该 Skill；
  • 如有相似 Skill，应加入误触发测试。

Plan Case

Section titled “Plan Case”

用于验证模型是否理解 Skill 的执行流程。

Plan Case 必须要求：

  • 不实际执行工具；
  • 不修改文件；
  • 只输出计划；
  • 明确使用哪个 Skill；
  • 明确步骤、输入、输出、成功/失败标准、失败路由。

备注

如果用户 Query 是：“xxxx”，你会怎么做？ 不要实际执行工具，不要修改文件，只输出计划。

Contract Case

Section titled “Contract Case”

用于验证输出契约。

重点检查：

  • 产物文件名；
  • 必填字段；
  • 字段取值范围；
  • Markdown 标题结构；
  • JSON Schema；
  • 是否可被下游 Agent 或 Hook 消费。

Edge Case

Section titled “Edge Case”

用于验证信息不足、输入冲突、文档缺失等场景。

要求模型：

  • 不编造缺失信息；
  • 明确标记缺失字段；
  • 可输出 BLOCKED 或 NEEDS_CONFIRMATION；
  • 给出下一步确认问题。

Failure Case

Section titled “Failure Case”

用于验证失败诊断和回流策略。

失败场景应要求输出：

  • 失败用例；
  • 期望结果；
  • 实际结果；
  • 失败证据；
  • 疑似原因；
  • 建议回流对象；
  • 下一步动作。
